DELIVERABLE 2 -> DELIVERABLE 3 HANDOFF LOG
STAT-946 Case Study 3

Purpose
This log is a project handoff note for the current state of the NBA post-rookie extension workflow.
It is written so that, in a new project conversation, an AI assistant can read this file together with the listed CSV outputs and immediately know:

1. which Deliverable 2 outputs already exist and are relevant for Deliverable 3,
2. what each file means,
3. how each file should be interpreted and used,
4. what Deliverable 3 work can start now,
5. what work must remain on hold because Deliverable 1 has not yet been fully integrated.

This log assumes the current project framing:
- Deliverable 1 = Performance Forecast
- Deliverable 2 = Player Archetype & Comparable Profile
- Deliverable 3 = Salary Decision Support

Current project logic
Deliverable 2 is already implemented as a hybrid archetype system:
- macro role from saved PCA + K-means artifacts,
- shot-style subtype from the ShotChartDetail spatial embedding pipeline,
- identity drift across Seasons 1-4,
- realistic comps from the drafted-player pool,
- optional Hall of Fame ceiling comps,
- and stakeholder-facing dossier tables/visuals.

Deliverable 3 is intended to combine:
- Year-5 contract value as salary-cap percentage,
- durability/risk adjustment,
- final extension recommendation.

At the current project stage, Deliverable 3 can already start in its salary-target and comp-market-context components.
However, the final recommendation layer is still incomplete because Deliverable 1 results are not yet fully integrated.

--------------------------------------------------------------------------------
A1. CORE DELIVERABLE 2 OUTPUTS
--------------------------------------------------------------------------------

1. archetype_macro_player_table.csv
Role:
- canonical macro-role table from the saved PCA + K-means clustering backbone.

Use for Deliverable 3:
- optional upstream reference if the assistant needs direct macro-role predictors,
- role-family interpretation,
- archetype scarcity / replaceability language,
- validating prototype-fit distance fields.

Interpretation:
- one row per player,
- contains macro cluster, macro archetype, own-cluster distance, cluster-distance predictors, and prototype ambiguity measures.
- this is the broad role taxonomy, not the final player identity by itself.

Why it matters:
- if Deliverable 3 needs role-family information directly rather than using only the final profile table, this is the backbone file.


2. shot_style_cluster_summary.csv
Role:
- dictionary-style summary of the five shot-style subtypes learned from the autoencoder embeddings and Gaussian mixture model.

Use for Deliverable 3:
- interpretation only,
- helps explain what a subtype label actually means in basketball language,
- useful if the assistant needs to convert subtype IDs or subtype names into front-office wording.

Interpretation:
- subtype-level, not player-level,
- includes subtype sizes, average membership probabilities, entropy, and descriptive style notes.

Why it matters:
- Deliverable 3 should not treat subtype labels as empty category names. This file provides the descriptive meaning.


3. shot_style_player_table.csv
Role:
- player-level shot-style subtype table.

Use for Deliverable 3:
- deeper diagnostic layer if the assistant needs subtype probability, entropy, total shot coverage, or embedding counts beyond the final profile table.

Interpretation:
- one row per player,
- contains dominant shot-style subtype, soft membership probability, entropy, total shot attempts covered, seasons with embeddings, and the player-level shot embeddings.

Why it matters:
- if Deliverable 3 wants to weight or discount a comp-market anchor based on subtype clarity, this is the file that contains the raw subtype-confidence information.


4. shot_style_player_season_table.csv
Role:
- season-level shot-style subtype assignments.

Use for Deliverable 3:
- only if the assistant needs to inspect or audit shot-style drift season by season.

Interpretation:
- one row per player-season,
- not meant to be the main salary input table,
- primarily a drift and trajectory audit file.

Why it matters:
- useful when a player’s identity-drift class seems suspicious and the assistant wants season-level evidence.


5. player_hybrid_archetype_table.csv
Role:
- fused macro role + shot-style subtype player table.

Use for Deliverable 3:
- optional intermediate table,
- useful if the assistant wants to rebuild or audit the final profile table.

Interpretation:
- one row per player,
- hybrid label = macro archetype + shot-style subtype.

Why it matters:
- this is the first fused identity table before drift and comps are layered in.


6. player_role_drift_summary.csv
Role:
- boxscore-role drift summary.

Use for Deliverable 3:
- diagnostic only,
- if the assistant wants to separate boxscore drift from shot-style drift.

Interpretation:
- endpoint displacement and total path length in standardized season-level boxscore role space.

Why it matters:
- not the main handoff file, but useful if a player’s drift class needs to be justified more explicitly.


7. player_shotstyle_drift_summary.csv
Role:
- shot-style drift summary.

Use for Deliverable 3:
- diagnostic only,
- use when the assistant needs to see how much a player’s shot behavior changed across Seasons 1-4.

Interpretation:
- endpoint displacement and total path length in shot-style embedding space.

Why it matters:
- useful for auditing whether a role change was really spatial/style-based rather than just boxscore-based.


8. player_identity_drift_table.csv
Role:
- fused player-level identity-drift table.

Use for Deliverable 3:
- direct input if the assistant wants drift metrics before the final profile table,
- useful when the final profile table is not available or needs to be rebuilt.

Interpretation:
- one row per player,
- contains macro role, subtype context, drift metrics, subtype changes, and final identity_drift_class.

Why it matters:
- this is the player-level drift output that feeds the final Deliverable 2 profile table.


9. realistic_comps.csv
Role:
- row-level realistic comparable-player table.

Use for Deliverable 3:
- critical input for comp-market anchor construction,
- needed to trace each focal player to specific realistic comps,
- needed if the assistant wants to rebuild or audit comp salary anchors.

Interpretation:
- multiple rows per focal player,
- each row is one realistic comp candidate,
- includes comp rank, similarity score, shot embedding distance, PCA distance, prototype-fit gap, drift difference, and macro-match indicator.

Why it matters:
- this is the row-level comp engine output, not just a summary.
- if a Block 2 salary anchor looks wrong, this file is the first place to inspect.


10. ceiling_comps_hof.csv
Role:
- optional Hall of Fame ceiling-comp table.

Use for Deliverable 3:
- interpretation only,
- upside analog only,
- should NOT be used as the primary pricing anchor.

Interpretation:
- one optional ceiling comp per player,
- only useful as aspirational context.

Why it matters:
- Deliverable 3 should clearly distinguish ceiling comps from realistic market comps.


11. player_comp_dossier_table.csv
Role:
- summarized comp-group dossier table.

Use for Deliverable 3:
- intermediate summary table if the assistant wants comp-group medians and plain-language comp interpretation before using the final profile table.

Interpretation:
- one row per player,
- contains realistic comp list,
- mean similarity score,
- median later-career comp-group points/minutes/rebounds/assists,
- optional ceiling comp,
- plain-language comp interpretation.

Why it matters:
- this is the summary version of realistic_comps.csv.
- it is easier to read, but it is not sufficient for salary-anchor auditing by itself.


12. final_player_archetype_profile_table.csv
Role:

Use for Deliverable 3:
- primary Deliverable 2 input file,
- first file the assistant should read when building Deliverable 3 Block 2,
- best single-table summary of the player’s current identity and comp neighborhood.

Interpretation:
- one row per player,
- includes:
  - macro archetype,
  - shot-style subtype,
  - hybrid label,
  - prototype-fit ambiguity,
  - own-cluster distance,
  - identity drift class,
  - realistic comp list,
  - realistic comp similarity mean,
  - optional ceiling comp,
  - comp-group later-career medians,
  - supporting shot-style explanation,
  - comp-based interpretation.

Why it matters:
- this is the cleanest handoff table from Deliverable 2 into Deliverable 3.
- if only one Deliverable 2 file is uploaded to a new conversation, this should be one of them.


13. final_player_archetype_case_study_table.csv

3. Deliverable 1 player-level forecast outputs
Specifically pending:
- player-level Years 5-7 performance predictions,
- projected floor/base/ceiling if uncertainty is modeled that way,
- or equivalent final forecast card outputs.

4. Sleeper / neutral / bust probabilities
Needed because:
- Deliverable 3 final recommendation is supposed to synthesize upside/downside risk, not just median value.

5. Uncertainty / confidence band outputs
Needed because:
- Deliverable 3 final recommendation is supposed to include uncertainty and avoid false precision.

6. Any validated baseline-vs-sequence model comparison results from Deliverable 1
Needed because:
- the final case-study logic should know whether the assistant is using baseline-only forecasts or a stronger hybrid forecast.

7. Final Deliverable 1 player-level table ready for merge
Ideal expected file structure from Deliverable 1 would include fields such as:
- PLAYER_ID
- PLAYER_NAME
- predicted_years5_7_points (or equivalent supported outcomes)
- predicted_years5_7_minutes
- predicted_years5_7_rebounds
- predicted_years5_7_assists
- sleeper_probability
- neutral_probability
- bust_probability
- uncertainty_band_low / base / high OR another explicit uncertainty structure
- model_support_or_confidence flag if used

Without those Deliverable 1 outputs, the assistant should NOT claim Deliverable 3 is fully complete.
